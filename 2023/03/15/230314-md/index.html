웹 접근성

웹 접근성 애플리케이션과 위젯

문제 상황

  • 웹은 빠른 속도로 변화하고 있습니다. 페이지 중심의 정적 사이트들은 점점 더 동적으로 변화하고 있으며, 데스크톱 웹 어플리케이션들이 JS와 AJAX를 중심으로 제작됩니다. 이러한 변화는 사용성과 반응형 향상에는 극적으로 도움을 주었지만, 스크린 리더와 같은 보조 기술(AT)을 사용하는 유저들에겐 접근성 격차로 인한 제약이 발생하는 리스크도 안고 있습니다.
  • 예전의 HTML4 명세가 나왔을 때는 Slider, Menu bar 등과 같은 클라이언트 사이드 위젯들의 설명이 제공되지 않았습니다. 하여 개발자들은 보통
    , 같은 것들을 활용하여 개발했습니다. 이에 따라 데스크톱 위젯들은 충분한 정보를 제공하지 못했습니다.
  • 동적 콘텐츠는 이유가 어떻든 스크린을 보지 못하는 사람들에게 문제가 되었습니다. 주식 시세 표시 위젯, progress bar 등과 같은 것들을 보조공학기술(AT)로 인식하지 못하는 경우가 많았습니다. ARIA는 이러한 간극을 줄이는데 그 의미가 있습니다.

ARIA(Accessible Rich Internet Applications)

  • ARIA는 W3C의 Web Accessibility Initiative에서 제작합니다.
  • 스크린 리더와 같은 보조기기에서 필요한 정보들을 추가하는 방법을 제공합니다.
  • ARIA는 마크업에서 특별한 속성을 추가하여 개발자들이 위젯의 디테일한 정보를 제공할 때 사용합니다.
  • 특히, 동적 웹 어플리케이션에서 찾을 수 있는 데스크톱 스타일 컨트롤과 표준 HTML 태그 사이에 있는 간극을 채우기 위해, ARIA는 친숙한 UI 위젯의 동작 상태(state)와 역할(role)에 대한 설명을 제공합니다.

ARIA - roles, states, properties

  • ARIA는 다른 타입의 속성 세개 roles, states, properties를 분할하여 정의합니다.

  • roles

    • Slider, Menu bar, dialog와 같은 HTML4에서 사용하지 못하는 위젯을 설명합니다.

  • properties

    • 드래그가 가능하다는 것이나, 요소가 필요하다는 것이나, 팝업이 뜨는 것과 같은 위젯의 특징에 대해 설명합니다.

  • state

    • 요소의 현재 상태에 대해 설명합니다. 보조기기에서 요소의 접근이 불가하거나, 숨겨져 있는 상태라는 것 등을 명시합니다.

State changes

  • aria-checked, aria-disabled, aria-grabbed

    • aria-* 속성을 사용하여 UI 위젯 요소의 상태를 표현하고, CSS attribute Selector를 활용하여 state 변경에 따른 시각적 변화를 구현해야합니다.

      • ```css
        li[aria-checked=”true”] {
        font-weight: bold;
        background-image: url(“images/dot.png”);
        background-repeat: no-repeat;
        background-position: 5px 10px;
        }

Visibility Changes

  • aria-hidden

    • aria-* 속성을 사용하여 UI 위젯 요소의 상태를 표현하고, CSS attribute Selector를 활용하여 state 변경에 따른 시각적 변화를 구현해야합니다.

      • ```css
        div.tooltip[aria-hidden=”true”] {
        display: none;
        }

Role Changes

  • ARIA를 통해 개발자는 부정확하거나 시맨틱을 제공하지 않는 요소에 대한 시맨틱 역할을 선언할 수 있습니다.

    • 예를 들어, 순서가 없는 목록을 사용하여 메뉴를 만들 때
        에는 menubar 역할을 부여하고 각
      • 에는 menuitem 역할을 부여해야 합니다.

    • 요소의 역할은 변경되지 않아야 합니다. 대신 원래 요소를 제거하고 새 역할을 가진 요소로 교체합니다.

    aria 속성 사용하기

    • ARIA 속성은 W3C에서 정의한 규격이 있으며, 이 규격에 따라 속성의 이름과 값이 정해져 있습니다. 예를 들어, aria-label 속성은 W3C에서 정의한 규격에 따라 해당 요소의 라벨을 지정하는 데 사용되며, aria-live 속성은 해당 요소의 상태를 실시간으로 알리는 역할을 합니다. 따라서, ARIA 속성을 사용할 때는 해당 속성의 규격을 확인하고, 그에 맞게 적절한 속성 이름과 값을 사용해야 합니다.

    • 다양한 aria-* 속성을 mdn에서 확인해볼 수 있습니다.

    • <ul id="fontMenu" class="menu" role="menu" aria-hidden="true">
  <li
    id="sans-serif"
    class="menu-item"
    role="menuitemradio"
    tabindex="-1"
    aria-controls="st1"
    aria-checked="true"
  >
    Sans-serif
  </li>
  <li
    id="serif"
    class="menu-item"
    role="menuitemradio"
    tabindex="-1"
    aria-controls="st1"
    aria-checked="false"
  >
    Serif
  </li>
  ...
</ul>

    role 속성 사용하기

    • role 속성은 WAI-ARIA(웹 접근성 증진을 위한 인터넷 애플리케이션) 사양에서 정의된 역할(role)을 나타내는 속성입니다. WAI-ARIA 사양은 웹 콘텐츠의 접근성을 개선하기 위해 만들어진 것으로, 이 중 역할(role)은 웹 페이지의 요소가 어떤 역할을 수행하는지 설명하는 역할을 합니다.

    • role 속성을 사용할 때는 WAI-ARIA 사양에서 정의된 역할을 사용하는 것이 좋습니다. 예를 들어 “button”이나 “checkbox”와 같은 역할을 가지는 요소에는 해당 역할을 나타내는 “role” 속성을 사용하는 것이 적합합니다.

    • 주의사항

      • 올바르지 못한 ARIA를 사용할 바엔 ARIA를 사용하지 않는 편이 좋습니다.
      • ARIA를 사용하기 전에 태그의 역할과 의미에 맞게 작성합니다.
      • 태그의 기본 의미를 중복해서 선언할 필요는 없습니다.
      • 페이지에서 사용하는 태그의 역할이 잘못된 ARIA를 선언하면 안됩니다.

    • html 각 태그별로 **암묵적으로 가지고 있는 role(역할)**이 있고, 각 태그별로 적용할 수 있는 역할도 있습니다.
